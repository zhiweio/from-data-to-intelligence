这一附录收集读者最常问的问题。如果你在阅读中产生疑问，先来这里看看是否已有答案。

## 架构与选型类

### Q1：这个架构适合小型团队吗？

**简答**：不太适合。这套架构（AWS 自组装 + Terraform + Glue/Step Functions + Redshift）是为中型企业、100TB+ 规模、多业务域、强合规场景设计的。它的复杂度来自"可治理、可演进"——小型团队（<5 人、<1TB、单业务域）用这套架构会过度工程。

**更适合小型团队的方案**：直接用 :simple-snowflake: Snowflake 或 :simple-databricks: Databricks 一体化平台，省去 IaC 和组件集成的运维负担。详见 [Ch 52](./52-架构师的复盘-取舍遗憾与主流对比.md) 的主流方案对比与选型建议。

!!! tip "引申"
    判断"是否需要这套架构"的简单标准：如果你的数据量 <1TB、源系统 <5 个、没有 GxP/PIPL 合规要求、团队 <5 人，优先选一体化平台。当规模或合规复杂度上升到"自组装的灵活性收益超过集成成本"时，再考虑本书的方案。

### Q2：为什么不直接用 Snowflake / Databricks？

**简答**：不是不想，是当时不能。项目启动时（四年前）Snowflake 和 Databricks 尚未入华提供大陆商用服务，数据驻留要求只能选 AWS China。这是历史约束，不是技术偏好。详见 [Ch 2](./02-从需求到蓝图：一个数据平台的诞生.md) 选型争论与 [Ch 3](./03-技术栈全景与预备知识.md) AWS China vs Global 差异。

**如果今天重新选型**：[Ch 52](./52-架构师的复盘-取舍遗憾与主流对比.md) 给出了"如果重来"的建议——一体化平台（A 类）和湖仓开源（C 类）是首选，本书方案（B 类自组装）是特定历史约束下的务实选择，但其中的架构思想（五层模型、配置驱动、治理体系）是方案无关的。

## 数据工程类

### Q3：如何迁移现有的 Airflow DAG 到这个平台？

**简答**：本书平台的编排是 Step Functions + EventBridge，不是 Airflow。但迁移思路通用：

1. **梳理 DAG 依赖**：把 Airflow DAG 的任务依赖关系映射为 Step Functions 状态机的节点拓扑（[Ch 10](./10-编排与调度设计-StepFunctions与EventBridge.md)、[Ch 33](./33-自研DAG调度器与任务编排.md)）
2. **任务实体化**：每个 Airflow Task 对应一个 Glue Job 或 Lambda，用配置驱动声明（[Ch 12](./12-配置驱动的任务模型.md)）
3. **调度迁移**：Airflow 的 schedule 迁移为 EventBridge 定时规则
4. **渐进迁移**：先迁非关键 DAG 验证，再分批迁关键 DAG

!!! warning "Trade-off"
    如果你已有成熟的 Airflow 团队和大量 DAG，不必强行迁到 Step Functions——Airflow 本身是优秀的编排工具。[Ch 52](./52-架构师的复盘-取舍遗憾与主流对比.md) 也指出"单线程 DAG"是遗憾，Airflow 的有界并发反而更成熟。迁移与否取决于"统一编排栈"与"迁移成本"的权衡。

### Q4：增量加载的水位管理有什么坑？

**简答**：主要三个坑，详见 [Ch 14](./14-数据库与JDBC连接器.md)：

1. **硬删除漏数据**：时间戳水位查不到被物理删除的行 → 定期全量校准兜底
2. **迟到数据**：源系统延迟导致 `updated_at` 落后于水位 → 回溯窗口（3 天）+ 分区幂等覆盖
3. **水位回退**：重跑任务时水位被覆盖 → 水位持久化在 DynamoDB，重跑前备份

## Agentic BI 类

### Q5：Agentic BI 的 LLM 成本怎么控制？

**简答**：三层控制，详见 [Ch 47](./47-评估-可观测与持续演进.md) 成本分析：

1. **缓存命中**（35%）：语义相似度 ≥0.92 的查询直接返回缓存 SQL，跳过 LLM 调用（[Ch 42](./42-Agent编排-LangGraph与状态机.md)）
2. **简单查询强路由**：术语绑定直接命中 metric 定义，跳过规划器（[Ch 40](./40-语义平面-三层治理与Git-YAML.md)）
3. **多 Provider 抽象**：简单查询用便宜模型，复杂查询才用强模型（[Ch 47](./47-评估-可观测与持续演进.md) Roadmap P1）

**量级参考**：日均 500 次查询 ≈ ¥15,000/月，远低于替代的传统 BI 人力成本。成本失控的信号是"所有查询都走完整 LLM 流程"——把它改成"简单走缓存/强路由、复杂才走完整流程"。

### Q6：Agentic BI 上线准确率只有 75%，业务不信任怎么办？

**简答**：这是 30 天磨合期的正常表现，不要追求上线即完美。关键是建立"用户反馈→语义资产修正→准确率提升"闭环——每条用户纠正进入 Correction Memory（[Ch 45](./45-记忆系统与工具使用.md)），转化为下次 few-shot。30/60/90 天递进复盘（75%→85%→93%+）是典型形态（[Ch 51](./51-价值度量与案例复盘.md)）。初期可先用"咨询模式"（[Ch 44](./44-五层SQL护栏与执行安全.md)）——只展示 SQL 不自动执行，让业务确认后再跑，建立信任后再开自动执行。

### Q7：Steiner 树查询规划太难，能不能直接让 LLM 推理 join？

**简答**：可以，但复杂查询准确率会显著下降。Steiner 树的价值正在于把"join 路径选择"从"LLM 猜"变为"算法算"——简单查询 LLM 推理够用，但多表 join + 聚合时 LLM 经常选错路径（鸿沟陷阱/扇出陷阱）。引入 Steiner 树后复杂查询准确率提升约 12%（[Ch 43](./43-语义查询规划器-Steiner树与代数改写.md)、[Ch 47](./47-评估-可观测与持续演进.md)）。如果团队暂时无力实现规划器，至少要先上五层护栏（[Ch 44](./44-五层SQL护栏与执行安全.md)）兜底。

## 治理类

### Q8：医药行业的 GxP/PIPL 合规，最小必要投入是什么？

**简答**：三件套，缺一不可，详见 [Ch 18](./18-数据脱敏与隐私治理.md)、[Ch 48](./48-安全-合规与治理.md)：

1. **数据驻留**：选 AWS China（cn-north-1），满足数据不出境
2. **字段级脱敏**：至少实现 mask/md5/aes_encrypt 三种策略 + 配置驱动（[Ch 18](./18-数据脱敏与隐私治理.md) 决策表）
3. **全链路审计**：审计日志 + RLS/CLS 权限控制（[Ch 48](./48-安全-合规与治理.md)）

合规不是事后补丁，是架构第一天的非功能需求——事后补的代价远高于一开始就嵌入。

## 基础设施 / Terraform 类

### Q9：Terraform state 锁死了怎么办？

**简答**：DynamoDB lock 偶发锁死（apply 崩溃未释放锁）。应急用 `terraform force-unlock <LOCK_ID>`，但需规范：① 两人确认（ChatOps 审批）；② 操作记录到 audit log；③ 事后排查锁死根因（通常是 apply 进程被 kill）。详见 [Ch 21](./21-Terraform架构总览.md)。**切勿**直接删 DynamoDB lock 表项——先确认没有其他 apply 真的在跑。

### Q10：如何升级 Terraform 模块版本？

**简答**：改业务仓的 `?ref=vX.Y.Z` 指针，走 DEV→QA→PROD 完整发布流（[Ch 24](./24-通用Terraform模块设计.md)）。Major 版本（破坏性变更）要：① 读模块 CHANGELOG 确认 breaking changes；② 在 DEV 跑 `terraform plan` 检查 diff；③ 准备回滚方案（指针指回旧版）。Minor/Patch 版本通常向后兼容，但仍需 QA 验证。

### Q11：Drift Detection 发现漂移了怎么办？

**简答**：周级 `terraform plan -detailed-exitcode` 检测到漂移（有人手动改了 AWS 资源），两种处理：① **手动修改是错误的**→提修复 PR 把 Terraform 代码改回与实际一致，或用 `terraform apply` 把资源改回代码状态；② **手动修改是必要的**→更新 Terraform 代码纳入管理。无论哪种，都要补 audit log 说明"为什么漂移"。详见 [Ch 28](./28-四类发布流.md)。

## 团队 / 组织类

### Q12：数据平台团队该怎么招人？

**简答**：平台需要四类角色——数据工程师（ETL/Spark）、IaC 工程师（Terraform/CI/CD）、平台运维（监控/排障/成本）、AI 工程师（Agentic BI，后期）。最难招的是"既懂数据工程又懂 IaC"的复合型人才——建议内部培养而非外部招聘：先招扎实的数据工程师，再补 IaC 技能。详见 [Ch 53](./53-致谢与团队.md) 团队结构。

### Q13：知识转移怎么度量？

**简答**：咨询式交付的终极目标是"把自己交付到不需要自己"（[Ch 53](./53-致谢与团队.md)）。度量指标：① 甲方团队独立解决的 issue 占比（目标 >70%）；② 甲方独立新增数据源的 lead time（目标 <1 周）；③ 文档完备度（每个流程有 runbook）。知识转移不是"培训完就结束"，而是"甲方能独立运营"才算完成。

## Agentic BI 类

### Q14：Agentic BI 用多家 LLM Provider 怎么管理？

**简答**：抽象一个 Provider 层，简单查询走便宜模型、复杂查询走强模型（[Ch 47](./47-评估-可观测与持续演进.md) Roadmap P1）。关键是统一接口——`llm.complete(prompt, model)` 屏蔽各家 API 差异，切换 Provider 只改配置不改代码。China 区域需注意 Provider 可用性（Bedrock 模型子集与 Global 不同，见 [Ch 3](./03-技术栈全景与预备知识.md)）。

### Q15：SQL 缓存的相似度阈值 0.92 怎么调？

**简答**：0.92 是准确率 vs 命中率的平衡点（[Ch 42](./42-Agent编排-LangGraph与状态机.md)）。调高（0.95）命中率下降但更安全；调低（0.85）命中率上升但可能返回不精确 SQL。建议：① 先用 0.92 跑一个月，统计缓存命中后用户是否纠正（纠正率高说明阈值太低）；② 按查询类型分级——简单查询阈值可低（0.88），复杂查询阈值要高（0.95）。

### Q16：HITL 审批会不会拖慢效率？

**简答**：只对高风险操作触发（DDL/大批量 DELETE/跨租户导出，[Ch 42](./42-Agent编排-LangGraph与状态机.md)），这类操作本就低频。关键是审批通道要顺畅——推 Slack/Teams，业务负责人手机可批，设超时自动拒绝（避免悬而未决）。HITL 不是"每次查询都审批"，是"少数高风险操作把关"，对整体效率影响 <5%。
